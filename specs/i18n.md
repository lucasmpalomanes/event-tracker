## 1. Overview

The app ships with a mix of English chrome and PT-BR domain copy. This spec
internationalizes the whole UI with **i18next**, supporting four languages:

| Locale  | Language              | Endonym (shown in selector) | Flag |
|---------|-----------------------|------------------------------|------|
| `pt-BR` | Brazilian Portuguese  | Português                    | 🇧🇷 |
| `en`    | English               | English                      | 🇬🇧 |
| `zh-CN` | Chinese (Simplified)  | 中文                          | 🇨🇳 |
| `bs`    | Bosnian               | Bosanski                     | 🇧🇦 |

A **language selector with flags** lives in the home page toolbar. The chosen
language is a per-browser preference (cookie) that applies to **every** page,
not just the home page.

`pt-BR` is the default and fallback language — the user base is Brazilian and
the money flow is Pix/BRL.

## 2. Goals & non-goals

### Goals
- All UI strings (labels, buttons, badges, empty states, validation messages,
  server-action error messages, `<title>`/description metadata) served from
  i18next resources in the four languages.
- Language selector with flag + endonym in the home page header, visible in
  both the logged-out hero and the logged-in toolbar.
- Locale-aware **date** formatting (`Intl.DateTimeFormat`).
- Correct pluralization per language via i18next's `Intl.PluralRules`
  integration (Bosnian has `one/few/other`; Chinese has a single form).
- Server components render already-translated HTML — no flash of untranslated
  content, no full resource bundles shipped to the client.

### Non-goals (this version)
- Translating **user-generated content** (event titles, descriptions,
  locations, budget item names) — stored and shown as typed.
- Locale-prefixed URLs (`/en/events/…`). The app is auth-gated with no SEO
  concern, and invite links are shared between users with different
  preferences — links must stay locale-neutral. See §3 and Resolved decisions.
- Per-account language stored in the database (cookie only; see Future work).
- Right-to-left support (none of the four languages needs it).
- Localizing **money formatting**: amounts are always rendered in the BRL
  convention ("R$ 1.234,56") in every UI language — see §5.
- Translating Supabase/Auth0/Mercado Pago error payloads passed through
  verbatim — only messages the app itself authors.

## 3. Architecture

### Libraries

- `i18next` — core.
- `react-i18next` — hooks (`useTranslation`) for client components.
- `i18next-resources-to-backend` — lazy-loads JSON resource files per
  language/namespace on both server and client.
- `negotiator` + `@formatjs/intl-localematcher` — `Accept-Language`
  negotiation, as recommended by the Next.js internationalization guide
  (`node_modules/next/dist/docs/01-app/02-guides/internationalization.md`).

### Locale resolution (no routing change)

The Next guide's primary pattern is a `/[lang]` path segment. We deliberately
**do not** adopt it (see Non-goals); instead the active locale is resolved
per request, server-side:

1. `NEXT_LOCALE` cookie, if set to a supported locale;
2. otherwise best match of the `Accept-Language` header against the supported
   list (`intl-localematcher`);
3. otherwise `pt-BR`.

Implemented as `resolveLocale()` in `lib/i18n/settings.ts` using `cookies()` /
`headers()` from `next/headers`. No change to `proxy.ts` — the app is already
fully dynamic (auth on every request), so reading headers costs nothing new.

### Server side

`lib/i18n/server.ts` exposes `getT(namespace)`: creates (and per-request
caches, via React `cache()`) an i18next instance initialized with the resolved
locale, and returns the `t` function. Server components and **server actions**
(`app/actions.ts` validation/error messages) both use it.

The root layout (`app/layout.tsx`) sets `<html lang={locale}>` from
`resolveLocale()` and renders localized `metadata`.

### Client side

A client `I18nProvider` (mounted in the root layout) receives from the server
the resolved locale plus the initial namespaces' resources as props, creates
the client i18next instance synchronously with them (no async load before
first paint, so server and client HTML match), and client components use
`useTranslation()` from `react-i18next`. Additional namespaces lazy-load via
`i18next-resources-to-backend`.

### Resource files

```
locales/
  pt-BR/{common,home,event,budget,payment}.json
  en/…   zh-CN/…   bs/…
```

- `common` — app name, log in/out, generic buttons (save, cancel, remove),
  shared errors, status badges (`open`/`closed`/`finalized`), holiday names
  (`lib/holidays.ts` — currently PT-BR only).
- `home` — home page: hero, events list, membership buttons, pending badge.
- `event` — event page header, Dates tab, calendar, voting, create/edit forms.
- `budget` — Budget tab (`event-budget.md` §6 copy).
- `payment` — payment card/board, activation form (`pix-payments.md` §7 copy).

Keys are semantic and nested (`events.pendingRequests`), never the English
sentence. Interpolation with `{{name}}`; counts use i18next plural suffixes
(`_one`, `_few`, `_other`) — e.g. the "N pending request(s)" badge, group
sizes ("Drinkers · 6 people"), day counts.

All four languages are authored **complete** in the migration PR; `pt-BR` is
`fallbackLng` for anything that slips through later.

## 4. Language selector

**Placement** — the home page toolbar:

- Logged in: in the existing header (`app/page.tsx`), left of the user name.
- Logged out: the hero screen gains a minimal top bar with just the selector,
  right-aligned — visitors should be able to switch before logging in.

**UI** — a shadcn/ui `Select` (compact trigger). The trigger shows only the
current flag; each option shows flag + endonym ("🇧🇷 Português", "🇬🇧 English",
"🇨🇳 中文", "🇧🇦 Bosanski"). Flags are four small **inline SVGs** bundled as a
`Flag` component — emoji flags render as letter pairs on Windows, so they are
not acceptable as the only rendering. The trigger carries an `aria-label`
("Idioma" / "Language" / …, itself translated).

**Behavior** — a client component. Selecting a language calls a server action
`setLocale(locale)` that validates the value, sets the `NEXT_LOCALE` cookie
(path `/`, max-age 1 year, `SameSite=Lax`), then the component calls
`router.refresh()` so the current server-rendered tree re-renders in the new
language. No navigation, no URL change; the whole app follows on subsequent
requests.

## 5. Formatting

- **Dates** — replace the hand-rolled `formatDay()` (DD/MM/YYYY) with
  `Intl.DateTimeFormat(locale)` helpers in `lib/format.ts`: numeric dates for
  ranges, and a long form ("9 de julho", "July 9", "7月9日", "9. juli") where
  the calendar/ranking shows friendly dates. Weekday/month names in the voting
  calendar come from `Intl` too, not from translation files.
- **Currency** — **not localized.** Amounts are always BRL and always
  formatted the Brazilian way ("R$ 1.234,56"), whatever the UI language:
  `formatBRL()` stays exactly as it is (`Intl.NumberFormat("pt-BR", …)`).
  Money values must look identical for every participant of an event.
- **Lists/joins** (e.g. names in the ranking tooltip) — `Intl.ListFormat`.

## 7. Resolved decisions

- **Cookie preference, not `/[lang]` routes.** Language is a per-user
  preference, not content identity: invite links are shared across users with
  different languages and must not encode one, and the app is auth-gated so
  there is no SEO to serve. *(2026-07-09)*
- **`pt-BR` is default and fallback** — the audience is Brazilian; `en` is
  the second language, not the source of truth (today's English strings get
  replaced, not kept as fallback). *(2026-07-09)*
- **Chinese = Simplified (`zh-CN`)**, UK flag for English, and inline **SVG
  flags** (emoji flags break on Windows). *(2026-07-09)*
- **Selector lives on the home toolbar only** (as requested); the preference
  itself is global. Other pages render in the chosen language but have no
  selector of their own. *(2026-07-09)*
- **Currency is always BRL, in BRL formatting** regardless of UI language —
  the charge rail is Pix, and every participant of an event should see the
  same "R$ 1.234,56" string. Number formatting does **not** localize.
  *(2026-07-09)*
